# Reasoning Token Support — Design Spec

**Jira:** OLS-3442 (follow-on from OLS-2475 spike)
**Date:** 2026-07-03
**Status:** Draft

## Problem

OLS has two gaps in reasoning token support:

**Gap 1 — Provider enablement:** Only the OpenAI provider configures reasoning output (via model-name detection for o-series/gpt-5). Azure OpenAI, vLLM (RHELAI/RHOAI), Vertex AI Gemini, Vertex AI Anthropic, and Bedrock do not request reasoning output from their models.

**Gap 2 — Reasoning dropped in streaming pipeline:** Even for the working OpenAI provider, `StreamChunkType.REASONING` chunks are streamed to the client but never accumulated into `response`. When `store_conversation_history` runs, it creates `AIMessage(content=response)` with only text content — reasoning is lost.

## Design Decisions

These decisions were made during brainstorming based on research into LangChain's reasoning abstractions, provider API behavior, and storage impact analysis.

### Config-driven, not model-name detection

The current OpenAI provider hard-codes model-name checks (`if "gpt-5" in self.model or self.model.startswith("o")`). This breaks as new models ship and doesn't generalize to other providers. Instead, reasoning enablement is an explicit per-model config decision made by the operator in the OLSConfig CR.

**Rationale:** Provider APIs return 400 errors if you send reasoning params to non-reasoning models. There is no reliable cross-provider API to auto-detect reasoning capability. The operator knows what model they deployed.

### Freeform `reasoningConfig` map

Each provider has different reasoning config keys, and those keys change across model generations within the same provider (e.g., Gemini 2.5 uses `thinking_budget`, Gemini 3 uses `thinking_level` — they're mutually exclusive at the API level). A typed CRD struct would require CRD changes every time a provider ships a new model generation.

Instead, `reasoningConfig` is a `map[string]interface{}` that the operator passes through to the service. Each provider's code reads the keys it understands and passes them to its LangChain adapter. Invalid keys produce a clear 400 error from the provider API.

### No cache schema changes

Research showed that providers returning large reasoning traces (OpenAI, DeepSeek) automatically strip prior-turn reasoning from conversation history. Providers that preserve it (newer Anthropic, Gemini) return summaries or bounded thinking content. The storage multiplier for the providers OLS uses today is typically 1.0x–1.5x.

Furthermore, OLS currently works with Gemini without thought signatures, and the existing plain-string `AIMessage` round-trips successfully. Cache schema changes are deferred until testing and evals show they're needed.

### vLLM requires a ChatOpenAI subclass

LangChain's `ChatOpenAI` explicitly drops non-standard response fields like `reasoning_content` from third-party providers. The LangChain team closed the feature request as "not planned." The established pattern (used by `langchain-deepseek`) is to subclass `BaseChatOpenAI` and override two methods. No new Python dependencies required.

## Configuration Model

### CRD (`ModelParametersSpec` in lightspeed-operator)

```go
type ModelParametersSpec struct {
    MaxTokensForResponse int                    `json:"maxTokensForResponse,omitempty"`
    ToolBudgetRatio      float64                `json:"toolBudgetRatio,omitempty"`
    ReasoningConfig      map[string]interface{} `json:"reasoningConfig,omitempty"` // NEW
}
```

### olsconfig.yaml (generated by operator)

```yaml
models:
  - name: o3
    parameters:
      max_tokens_for_response: 4096
      reasoning_config:
        effort: medium
        summary: concise
```

### Service (`ModelParameters` in config.py)

```python
reasoning_config: Optional[dict[str, Any]] = None
```

The existing `reasoning_effort`, `reasoning_summary`, and `verbosity` fields on `ModelParameters` are removed — replaced by the general-purpose `reasoning_config` dict.

## Provider `reasoningConfig` Reference

This table documents the valid `reasoningConfig` keys for each provider and must be included in release notes and user-facing documentation.

| Provider | `reasoningConfig` contents | Service applies to |
|---|---|---|
| OpenAI | `{effort: medium, summary: concise, verbosity: low}` | `reasoning=` + `verbosity=` params on `ChatOpenAI` |
| Azure OpenAI | `{effort: medium, summary: concise, verbosity: low}` | `reasoning=` + `verbosity=` params on `AzureChatOpenAI` |
| Gemini 3 (Vertex) | `{thinking_level: high, include_thoughts: true}` | kwargs on `ChatGoogleGenerativeAI` |
| Gemini 2.5 (Vertex) | `{thinking_budget: 4096, include_thoughts: true}` | kwargs on `ChatGoogleGenerativeAI` |
| Anthropic (Vertex) | `{type: adaptive, display: summarized}` | `thinking=` param on `ChatAnthropicVertex` |
| Bedrock (Anthropic models) | `{type: adaptive}` | kwargs on `ChatBedrockConverse` |
| Bedrock (OpenAI models) | `{effort: medium, summary: concise}` | `reasoning=` param on `ChatOpenAI` |
| vLLM (RHELAI) | `{enabled: true}` | Triggers `ChatVLLMReasoning` subclass |
| vLLM (RHOAI) | `{enabled: true}` | Triggers `ChatVLLMReasoning` subclass |

**When `reasoningConfig` is absent:** The provider behaves as today — no reasoning params are sent, standard temperature/top_p/frequency_penalty defaults apply.

**When `reasoningConfig` is present on a non-reasoning model:** The provider API returns a 400 error. OLS does not validate model-reasoning compatibility — the operator is responsible for correct configuration.

## Provider Changes

### OpenAI (`openai.py`)

Remove model-name detection (`if "gpt-5" in self.model or self.model.startswith("o")`). If `reasoning_config` is present in model parameters, pass it as the `reasoning` dict to `ChatOpenAI` and skip temperature/top_p/frequency_penalty. If absent, apply current defaults.

### Azure OpenAI (`azure_openai.py`)

Same pattern as OpenAI. Currently skips temperature for o-series (model-name check) but never passes `reasoning`. With this change: if `reasoning_config` is present, pass as `reasoning` dict and skip temperature. If absent, apply defaults.

### Gemini (`google_vertex.py` — `GoogleVertex`)

If `reasoning_config` is present, unpack it as kwargs (`thinking_level`, `thinking_budget`, `include_thoughts`, etc.) into the `ChatGoogleGenerativeAI` constructor.

### Anthropic via Vertex (`google_vertex.py` — `GoogleVertexAnthropic`)

If `reasoning_config` is present, pass it as the `thinking` dict to `ChatAnthropicVertex`.

### Bedrock (`bedrock.py`)

Two existing code paths:
- **Anthropic models** (`ChatBedrockConverse`): If `reasoning_config` is present, pass via `additionalModelRequestFields` or the `thinking` parameter (exact interface depends on `langchain-aws` version — verify during implementation).
- **OpenAI models** (`ChatOpenAI` with Responses API): If `reasoning_config` is present, pass as `reasoning` dict.

### vLLM (`rhelai_vllm.py`, `rhoai_vllm.py`)

If `reasoning_config` is present and has `enabled: true`, instantiate `ChatVLLMReasoning` instead of `ChatOpenAI`. Otherwise use `ChatOpenAI` as today.

### Parameter validation (`provider.py`)

Add `reasoning` to `OpenAIParameters` and `AzureOpenAIParameters`. Add `thinking_level`, `thinking_budget`, `include_thoughts` to `GoogleVertexParameters`. Add `thinking` to `GoogleVertexAnthropicParameters`. Add `BedrockParameters` set. Add `reasoning` to `RHELAIVLLMParameters` and `RHOAIVLLMParameters`.

## vLLM Reasoning Subclass

New file: `ols/src/llms/providers/vllm_reasoning.py`

`ChatVLLMReasoning` subclasses `BaseChatOpenAI` (same base as `ChatDeepSeek`) and overrides two methods:

1. **`_create_chat_result()`** — non-streaming. Calls `super()`, extracts `reasoning_content` or `reasoning` from the raw `openai.BaseModel` response (which preserves unknown fields via Pydantic `extra="allow"`), stores in `message.additional_kwargs["reasoning_content"]`.

2. **`_convert_chunk_to_generation_chunk()`** — streaming. Calls `super()`, extracts reasoning from `chunk["choices"][0]["delta"]`, stores in `message.additional_kwargs["reasoning_content"]`.

Both field names are checked to handle the vLLM rename (RFC #27755). Output is normalized to `additional_kwargs["reasoning_content"]` regardless of source field.

No new Python dependencies. ~145 lines including docstrings. A reference implementation is attached to the Jira ticket as a comment.

## Streaming Pipeline Changes

### `llm_execution_agent.py` — `_collect_round_llm_chunks()`

The existing `match chunk.content` block (line ~588) handles `str` and `list` content. The `list` path already dispatches `StreamChunkType.REASONING` via `_streamed_chunks_from_list_content()` — this works for OpenAI, Gemini, and Anthropic where reasoning arrives as `content_blocks`.

For vLLM, reasoning arrives via `chunk.additional_kwargs["reasoning_content"]`. Add a check after the existing content dispatch: if `additional_kwargs` has `reasoning_content`, yield `StreamedChunk(type=StreamChunkType.REASONING, text=...)`.

### `streaming_ols.py` — reasoning accumulation

Line 454: `response += item.text` only runs for `StreamChunkType.TEXT`. Add accumulation for `StreamChunkType.REASONING` so reasoning text is included in the `response` string passed to `store_conversation_history`.

## Out of Scope

- **Cache schema changes** — `store_conversation_history` continues to store `AIMessage(content=response)` as a plain string. No structured reasoning block storage. Revisit based on future evals.
- **DeepSeek `<think>` tag parsing** — vLLM models that emit reasoning as `<think>...</think>` inside text content (rather than the structured `reasoning_content` field) require pipeline classification work tracked separately.
- **Model-capability auto-detection** — No provider API queries to discover reasoning support. Operator declares it via `reasoningConfig`.
- **Prior-turn reasoning stripping** — OLS does not selectively strip old reasoning from cache. Providers handle this themselves (OpenAI/DeepSeek strip automatically; Anthropic/Gemini preserve).
- **Console UI changes** — The console already renders `reasoning` streaming events correctly.
